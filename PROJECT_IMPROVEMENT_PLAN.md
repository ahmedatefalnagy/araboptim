1. تهيئة Git repository للمشروع (`git init`) مع `.gitignore` سليم.
2. عمل commit أولي بالحالة الحالية ("baseline").
3. أخذ نسخة احتياطية من قاعدة البيانات الحالية.
4. التأكد من تشغيل المشروع محلياً (`php artisan serve` + `npm run dev`) قبل وبعد كل مرحلة.

**المخرجات:** Repo نظيف + baseline commit + DB backup.

## المرحلة 1 — إصلاح الأخطاء الحرجة (Bugs)

### 1.1 حذف الموظف (HRController::destroyEmployee)
**المشكلة:** الموظف مرتبط كـ سائق في `vehicles.driver_id` و `trips.driver_id` (والأخير NOT NULL ومربوط بسجلات مالية)، فالحذف بيفشل بـ FK constraint والرسالة بتظهر بعد فشل الـ exception فقط.

**الحل (حسب قرارك — منع + رسالة + تعطيل):**
- فحص العلاقات **قبل** محاولة الحذف (vehicles, trips, payrolls, advances, user account).
- لو فيه أي ارتباط → منع الحذف وإرجاع رسالة عربية واضحة تذكر نوع الارتباط بالتحديد.
- لو لا يوجد ارتباط → حذف آمن داخل `DB::transaction`.
- التأكد من ظهور زر **"تعطيل"** (toggleEmployeeStatus موجود بالفعل) بشكل بارز في صفحة الموظفين كبديل.
- (تحسين) إضافة `SoftDeletes` اختيارياً لاحقاً للأرشفة بدل الإخفاء بالحالة فقط.

**ملفات:** `app/Http/Controllers/HRController.php`، `resources/js/Pages/HR/Employees.jsx`.

### 1.2 طريقة الدفع "بنك" في الفاتورة المجمعة
**المشكلة:** `payment_mode` يقبل `cash,credit` فقط — لا يوجد `bank`.

**الحل:**
- Backend: تعديل التحقق إلى `in:cash,credit,bank` في `store` و `update`، وضبط منطق `payment_account_id` ليشمل `bank`.
- Frontend: إضافة زر/خيار **"تحويل بنكي (بنك)"** في `Invoices/Create.jsx` + إظهار اختيار الحساب البنكي عند اختياره.
- مراجعة قيد اليومية الناتج ليُسجّل على الحساب البنكي الصحيح.

**ملفات:** `app/Http/Controllers/InvoiceController.php` (سطور ~102، ~312)، `resources/js/Pages/Invoices/Create.jsx`.

### 1.3 فحص شامل لبقية النماذج
- مراجعة كل عمليات الحذف (Contacts, Items, Accounts...) لنفس مشكلة الـ FK.
- مراجعة التحقق (validation) الناقص في الـ Controllers.

## المرحلة 2 — توحيد وتقوية الـ PDF (التحويل إلى mPDF)

**لماذا mPDF؟** يدعم تشكيل العربية والاتجاه RTL أصلياً، فنتخلص من الحيل الهشّة الحالية:
- `PdfHelper::fixArabic` / `utf8Glyphs($text, 100, ...)` (يقطع الأسماء الطويلة عند 100 حرف).
- `PdfHelper::fixArray` (يحوّل الكائنات لمصفوفات ويكسر العلاقات غير المُحمّلة).
- خدعة `dir="ltr" + text-align:right`.

**الخطوات:**
1. تثبيت `mpdf/mpdf` (وإزالة الاعتماد على `barryvdh/laravel-dompdf` تدريجياً).
2. إنشاء **خدمة موحّدة** `App\Services\PdfService` تتولى توليد كل الـ PDF (سندات، فواتير، تقارير) بإعدادات RTL وخط Amiri مركزية في مكان واحد.
3. تنظيف قوالب Blade في `resources/views/pdfs/`:
   - حذف خدع الاتجاه واستخدام `dir="rtl"` طبيعي.
   - استخدام الكائنات مباشرة مع null-safety (`$voucher->contact?->name`) بدل مصفوفات `fixArray`.
   - layout أساسي موحّد (`pdf_master`) لكل المستندات.
4. نقل بيانات الشركة (الاسم، الرقم الضريبي، السجل، اللوجو) من الكود الـ hard-coded إلى **جدول Settings**.
5. التأكد من سلامة كل سند:
   - سند قبض / صرف / صرف مصروف (voucher)
   - الفاتورة الضريبية + فاتورة الشاحنات (مع QR لـ ZATCA)
   - سند تسليم / سند استلام بضاعة
6. اختبار توليد كل نوع PDF ببيانات حقيقية والتأكد من عدم وجود أخطاء عربية أو تخطيط.

**ملفات:** `app/Services/PdfService.php` (جديد)، `app/Http/Controllers/PdfController.php`، `resources/views/pdfs/*`، `app/Helpers/PdfHelper.php` (تبسيط/إزالة).

## المرحلة 3 — إعادة الهيكلة والكود النظيف (قابلية التوسع وإعادة الاستخدام)

الهدف: بنية احترافية مثل المشاريع الكبرى.

1. **طبقة الخدمات (Service / Action Layer):** نقل منطق الأعمال الثقيل من الـ Controllers (خاصة HRController و InvoiceController و TripController) إلى Services/Actions مخصّصة. حالياً معظم المنطق داخل الـ controllers.
2. **Form Requests:** استبدال `$request->validate()` المباشر بـ FormRequest classes منظمة (موجود جزء بسيط، نعمّمه).
3. **API Resources:** توحيد شكل البيانات المُرسلة للواجهة عبر Eloquent API Resources.
4. **Enums:** تحويل القيم الثابتة (أنواع الفواتير، طرق الدفع، حالات الموظف/الرحلة) إلى PHP 8.1 Enums بدل النصوص المتناثرة.
5. **توحيد الثوابت:** إزالة الـ magic strings المتكررة (`'cash'`, `'credit'`, `'bank'`...).
6. **تنظيف الجذر:** نقل/أرشفة كل سكربتات المرة الواحدة (`analyze_*`, `fix_*`, `scratch_*`, `import_*`, ملفات `.sql` و`.xlsx`) إلى مجلد `scratch/` خارج الـ deploy أو حذفها. حالياً ~80 ملف في الجذر تلوّث المشروع.
7. **معايير الكود:** تطبيق Laravel Pint (موجود في require-dev) لتوحيد ستايل PHP.

## المرحلة 7 — الربط مع ZATCA الفوترة الإلكترونية (المرحلة الثانية) [اختيارية/مستقلة]

الحالة الحالية: `ZatcaService` ينفّذ **المرحلة الأولى فقط** (QR بصيغة TLV). الـ XML الحالي مجرد stub، ولا يوجد توقيع رقمي ولا ربط API.

**النطاق المطلوب للمرحلة الثانية:**
1. توليد XML متوافق UBL 2.1 (فاتورة قياسية B2B + مبسّطة B2C).
2. الختم التشفيري (توقيع ECDSA) باستخدام شهادة CSID من ZATCA.
3. تدفق التسجيل (Onboarding): CSR → شهادة امتثال (CCSID) → فحوصات → شهادة إنتاج (PCSID).
4. QR تشفيري + سلسلة هاش الفواتير (PIH).
5. تدفق Clearance (B2B) و Reporting (B2C) عبر منصة فاتورة (sandbox ثم production).

**متطلبات من العميل (غير برمجية):** حساب بوابة فاتورة + OTP لإصدار الشهادة + تأكيد إلزام المنشأة بالمرحلة الثانية.

**ملاحظة تصميمية:** يتم تصميم `PdfService` وبنية الـ XML في المرحلة 2 بحيث تكون **جاهزة لاستيعاب المرحلة الثانية** لاحقاً دون إعادة عمل. يُفضّل استخدام مكتبة PHP مخصّصة لـ ZATCA (مثل `salla/zatca`) لتسريع التوقيع والـ UBL.

## ملاحظات ومخاطر

- المشروع يحتوي بيانات مالية حقيقية → أي تعديل على قاعدة البيانات يتطلب نسخة احتياطية أولاً.
